What becomes more complicated however, is the ways in which we store the information in a database, as well as how we access that information. As you might image, there are a number of ways in which we can organize data, and with that comes even more ways to extract/access that data. As technology in general continues to evolve, so too do our methods for using databases as efficiently as possible. Enter our main contenders: SQL vs. NoSQL.

------------------------

Relational Database Management

To understand exactly what SQL is, you must first understand what a Relational Database Management System (RDBMS) is and how it funtions. Relational databases store data in a very particular away, which in turn allows for very specific methods in which we can manipulate/access that data. An RDBMS stores information in tables, which consist of rows and columns that contain unique characteristics of a particular object. These tables are connected to one another through a RELATIONAL element, most often a numbered ID row called a primary key. Picture a table containing customer contact information, and another table which contains customer order information for the same group of customers. The primary key which connects these two tables together and allows us to combine their information would most likely be a "CUSTOMER ID" that is unique to each customer.

So where does SQL come in? SQL, or Structured Query Language is the language we use to access relational databases. It just so happens that over time, the acronym "SQL" has become a reference term for all relational databases in general. Hence you will occasionally see developers reference an RDBMS as a "SQL database system". More common SQL database systems include: Oracle, MySQL, and SQLite

Some pros and cons of SQL databases:

+ They are commonly used by most developers, and are supported by multiple vendors

+ With their wide usage, there are a multitude of resources available for creating/maintaining these systems

- As the data contained within an SQL database system continues to grow, the system itself begins to perform slower and slower

- They aren't very agile when it comes to application construction in that they must operate within the confines of table design


------------------------

Non-Relational Database Management

This now brings us to a quick and upcoming alternative to relational databases: NON-relational databases. These can be more difficult to understand, in that non-relational DBs come in a variety of forms, and are not as simple to visualize as say, a table. NoSQL databases, or Not Only SQL databases can store information in multiple forms such as key-value pair structures, graphs, or documents. With this greater flexibility in design comes an added agility for application construction, which is why the battle between SQL and NoSQL has continued to intensify in recent years.

As mentioned above, an inherent weakness of more traditional SQL databases is that their performance declines as the information contained within them grows. As you might imagine, this has led to some big name tech companies seeking more dynamic alternatives. In particular, Google has developed its own NoSQLDBS called BigTable, which can handle gigantic amounts of information (other NoSQL systems include MongoDB, Revis, and RavenDB).

Some pros and cons of non-relational database systems:

+ They can better handle large amounts of information in that addtional servers can be easily added on to share the workload

+ They are more dynamic and scalable, which can be a critical benefit depending on the situation

- There aren't as many resources availabe in maintaining these systems, as they are relatively new and not as popular.


------------------------

While the argument over which of these systems is more "beneficial" to developers will go on for some time, my initial research makes me believe that there may never be a definitive victor in this battle. Technology continues to change and adapt over time, and our options for data storage/manipulation will undoubtedly do the same. My inexperienced opinion is that each of these systems has unique advantages over the other, depeninding entirely upon the situation in which they are being used. But by all means, do some additional research and decide for yourself!
