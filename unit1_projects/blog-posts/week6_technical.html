Classes vs Modules





In my blog post from Week 5, you read all about Classes: how they're created, how they're used, and what they're good for. This week, I'd like to dive into another critical element of the Ruby programming language: modules.

------------------------

What is a Module?

As you're well aware of by now, Ruby gives us a large number of "tools" we can use to write our programs. These tools serve a variety of different purproses, and are only used when necessary. Where are tools kept? A TOOLBOX! That's where modules step in to our program. The reason for storing this info for later, is that it is much faster to simply call a module when necessary, rather than having all of the methods/constants of a module written into our program. This not only keeps the program running faster, but it also keeps the code in our program as clean as possible.What is stored in a module is largely up to the programmer, but they usually store methods and constants.

You already know what methods are, but what is a constant? A constant is like a variable, except that it maintains a constant value throughout a program. Because modules don't typically contain things that change, constants are well suited to be included in modules.

------------------------

What Makes Classes and Modules the Same?

Despite their differences, how are modules and classes similar? First, a programmer can create their very own unique module just as they would a class. They are each declared in much the same way, with the module/class name being written in CamelCase:

Class Declaration: class NameOfClass

Module Declaration: module NameOfModule

Modules are also similar to classes in that they contain information that can be used throughout a program (whenever called). Also, just as there are several built-in classes in Ruby such as the Object Class, Ruby also has built-in modules such as the Math Module. These built-in modules contain common CONSTANTS and methods that are often used in various programs.



------------------------

What Makes Them Different?

Now that you have a basic understanding of their similarities, consider their differences:

Where as a class is typically used to create instances of that class, modules are unable to create instances of themselves. By definition, an instance often contains characteristics/attributes that can CHANGE, based on information provided by the user pertaining to that specific instance. Hence why modules do not have this ability. Their intent is to store information that will NOT change, and is further reason why a module will not typically contain a variable. Variables can be modified throughout a program, whereas a CONSTANT (such as PI) should remain the same regardless of what it is being used for.

So if I can't create an instance of a module, how do I access it? Very simple. Ruby has us use either "include" or "require" to grant us access to the constants and methods contained within a module. When a module is pulled in for usage within a class, this is referred to as a mix-in.

Another distinct difference between modules and classes is that modules cannot contain sub-classes (or "sub-modules"), nor can they inherit from another module. With that said, classes pseudo-inherit from modules when they are mixed-in as mentioned above.

------------------------
