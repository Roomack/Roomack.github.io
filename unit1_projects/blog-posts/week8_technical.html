Getting to Know Ruby

Whenever you first meet someone, a popular first question to ask of that person is, "Where are you from?" To understand someone's (something's) origin is to better understand that person/thing itself. With that, Rails is a framework that is written in the Ruby programming language. Just in case you're unfamiliar with Ruby, it is an object-oriented, general-purpose programming language. Created over 20 years ago by Yukihiro Matsumoto, Ruby's popularity has grown immensely and is now one of the top 10 programming languages out there with regard to web programming. It is a dynamic language, popular because it can accomplish tasks using less code as compared to its programming cousins (Java, C++, etc.), and it can be learned more quickly than other languages as well.

Enter: Rails



So where does Ruby tie in with all of this? According to the RailsApp Project, Rails is a "software library that extends the Ruby programming language", or "software code that is added to the Ruby programming language" (hence the often used phrase "Ruby on Rails"). With Ruby's already existent awesomeness, using Rails to "extend" its greatness opens many doors with regard to our ability to build equally awesome websites! Just think of Ruby on Rails as Ruby on Steroids!

As I stated above, Ruby on Rails is used in the development of web applications. How does a framework benefit a developer as they're creating a new app? I'm a big fan of analogies, so let's use a slightly crude one to orchestrate the benefit of Rails. Imagine you are wanting to build a new deck on the side of your house. Before getting started, you'd need to draw up plans, purchase and organize your tools, and hire some good help (or borrow a brother-in-law) to help with the heavy lifting. Wouldn't it be great if you could have all of those steps done FOR YOU? That's what Rails accomplishes for a web developer. It helps organize your development tools (Ruby, JavaScript, HTML, CSS), gives you a loose framework to start with, and also helps you out with some of the heavy lifting along the way!

More technically speaking, it is "opinionated" software that encourages the developer to abide by certain suggestions and conventions contained within the Rails API (Application Programming Interface). This API can be viewed as a rulebook which standardizes the use of Rails. This standardization allows for better collaboration between developers, and paves the way for more focused resources for maintaining and improving the framework itself.

The MVC Pattern

Let's take another look at the Rails description I gave you earlier:



The Model - View - WHO? To truly understand Rails, you'll need to also understand (recurring theme here, right?) the Model-View-Controller (MVC) pattern. MVC essentialls splits up your application intro three sections, or "layers", giving each individual sections it's own specific responsibility.

The model is essentially the central hub for all of the application's data and rules.

The view refers to any form of output to the user.

The controller is responsible for accepting input from the user and converting into commands for the model/view.

Based on the graphic below, you can see that these three components interact with one another as the application is being used, and allow the app to change with the user's direction/input. As you can see, this pattern is critical to the operation of the application, and explains why so many frameworks (like Rails) are geared towards MVC.
